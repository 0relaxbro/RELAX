Step 1 — Create + Initialize Candy Machine

The connected wallet must be the collection's update authority — InitializeV2 makes it sign so the Candy Machine can approve itself as a collection delegate. If it isn't, expect an IncorrectCollectionAuthority or a Token Metadata CPI failure, not a size error.

Step 2 — AddConfigLines (test batch)

Loads every item (Step 1's "items available") in as many batches as the ~1232-byte transaction limit needs. This is not optional: the Candy Machine refuses to mint at all until items_loaded == items_available (error 6020, NotFullyLoaded), so a partial load makes Step 4 fail no matter how correct its wiring is.

(not run yet)

Step 3 — Candy Guard: initialize + wrap

Creates a CandyGuard with a single group "WAVE1" (AllowList + MintLimit(1) + StartDate=now, matching the badge-wave design in the migration plan), then wraps it onto the Candy Machine from Step 1 so minting only works through the guard from here on.

The connected wallet must be one of these lines (or the list left empty) for Step 4 to find a valid proof for it. Every real winner's wallet goes here, one address per line, exactly as copied from their own wallet — extra whitespace is trimmed automatically, blank lines are ignored.

Metaplex's own SDK builds the tree from base58 address strings (its getMerkleProof takes a base58 string), so that is the default. If route comes back with AddressNotFoundInAllowedList, the on-chain guard hashes raw bytes instead — switch this, then re-run Step 3 (the root is baked into the guard, so a new guard is needed) and Step 4.
